## 1. What we're building

Tether is a mobile app (iOS first, Android later) that lets a developer talk to a coding agent — Claude Code, opencode, Gemini CLI, etc. — running on their own remote machine (home dev box, cloud VM, work workstation reachable via Tailscale).

The app is an **ACP (Agent Client Protocol) client over SSH**. It does not run an agent locally. It does not call any LLM API directly. It opens an SSH channel to the user's machine, executes the agent in ACP mode (e.g. `opencode acp`), and speaks JSON-RPC 2.0 over that pipe. The phone becomes a thin, well-designed front-end for an agent that lives on the user's hardware.

The product wager: developers increasingly want to drive coding agents from anywhere, but they don't want to hand their code, credentials, or compute to a third-party SaaS. They have a machine. They have Tailscale or SSH set up. They just don't have a usable interface from their phone.

## 2. Why now

- ACP exists and is stabilizing (Zed-originated, adopted by opencode, Claude Code via adapter, Gemini CLI experimental).
- Agent-driven coding is mainstream enough that "running an agent on your dev box" is a normal workflow.
- Tailscale + iOS makes always-on private connectivity to a personal machine genuinely easy for the first time.
- No incumbent on mobile. Blink Shell is the closest neighbor and it is a terminal, not an agent client.

## 3. Non-goals (beta)

State these clearly so we don't drift:

- Not a general SSH client or terminal emulator.
- Not a code editor. We render diffs and file content; we do not let users hand-edit files in-app.
- No on-device LLM, no proxying to Anthropic/OpenAI/Google APIs ourselves.
- No collaborative/multi-user sessions.
- No agent-side installer or auto-config. Users bring their own working agent.
- Android, iPad-optimized layout, and Apple Watch are post-beta.

## 4. Target user

Primary persona for beta: solo developer, 3+ years experience, already uses a coding agent daily on a desktop, has at least one always-on machine (home server, cloud VM, work box reachable via VPN/Tailscale), comfortable with SSH keys.

We are explicitly **not** designing for users who need us to host anything, manage agents for them, or abstract away SSH.

## 5. User journeys

### 5.1 First-run setup

1. Install app, open it.
2. Add a host: nickname, hostname (e.g. `devbox.tail-scale.ts.net`), username, auth method (SSH key from Keychain, or password — key strongly preferred).
3. Specify the agent command (default suggestions: `opencode acp`, `claude-code-acp`, `gemini --experimental-acp`).
4. Tap Connect. App opens SSH, execs agent, completes ACP `initialize` handshake, shows green status.
5. Land on empty chat screen for a new session.

### 5.2 Daily use — "ask agent something on the train"

1. Open app. Last session resumes (see §7.3 on session persistence).
2. Type or dictate a prompt. Hit send.
3. Assistant tokens stream in. Tool calls appear as cards as the agent works.
4. Permission prompt slides up when agent wants to run a command or write a file. User approves, rejects, or approves-with-edit.
5. Phone locks. Train goes through a tunnel. User reopens app 10 minutes later — session is still there, any output that arrived while disconnected is replayed and visible.

### 5.3 Reviewing a diff

1. Agent proposes an edit. A diff card appears inline in the conversation.
2. Tap to expand → unified diff view, syntax highlighted, with surrounding context expandable.
3. Approve / reject / "approve all in this session" toggle.

## 6. Scope for beta

In scope:

- iOS 17+ (iPhone only; iPad runs but is not optimized).
- One agent connection at a time per session; multiple saved hosts.
- Chat with streaming responses.
- Tool call rendering: file reads (collapsed, expandable), file writes/edits (diff view), shell commands (command + output).
- Permission prompts: allow / deny / allow-for-session.
- Session persistence across app suspension and brief network drops (§7.3).
- SSH key auth via iOS Keychain; password auth as fallback.
- Markdown + syntax highlighting in messages.
- Dark mode (default) and light mode.

Out of scope for beta:

- File browsing of the remote machine outside what the agent surfaces.
- In-app editing of agent-proposed diffs (approve or reject only).
- Voice input (post-beta; iOS dictation is acceptable for now).
- Multiple concurrent sessions visible side-by-side.
- Agent-to-agent or multi-host workflows.

## 7. Technical architecture

### 7.1 Stack

- **Framework:** React Native, bare (not Expo). Expo dev client adds friction for the native SSH module we have to ship.
- **Language:** TypeScript strict mode on the JS side. Swift for the iOS native module.
- **State:** Zustand. One store per active session; a root store for hosts and app-level state.
- **Lists:** `@shopify/flash-list` everywhere a message list, diff, or tool-call log lives. `FlatList` perf is not adequate for streamed tokens.
- **Markdown:** `react-native-markdown-display` with custom renderers for code blocks.
- **Syntax highlighting:** `react-native-syntax-highlighter` with Prism. Lazy-load grammars.
- **Secrets:** `react-native-keychain` for SSH keys and host passwords. Never written to JS-accessible storage.

### 7.2 SSH + protocol layer

- **SSH:** native module wrapping `NMSSH` (libssh2-based) on iOS. Exposes an event-emitter interface to JS: `onStdout`, `onStderr`, `onExit`, plus a `write` method.
  - Do **not** request a PTY. Pass `-T` semantics. PTY allocation mangles JSON-RPC framing.
  - Disable any remote shell init by execing the agent command directly via `channel.exec()`, not by opening a login shell.
- **Framing:** newline-delimited JSON-RPC 2.0. ACP currently uses LSP-style `Content-Length` headers in some implementations and bare NDJSON in others — the framer must detect and handle both. Test against opencode and claude-code-acp; they differ.
- **JSON-RPC client:** ~200 LoC of TypeScript. Outbound requests get incrementing ids, return promises resolved on matching response. Notifications dispatched to listeners by method name. No third-party library — we need to debug this and an opaque dep will fight us.
- **ACP version:** target the latest stable schema as of beta start. Pin per-agent quirks in an adapter layer (`adapters/opencode.ts`, `adapters/claude.ts`) so divergence doesn't leak into core.

### 7.3 Session persistence (the load-bearing problem)

iOS will suspend the app within ~30s of backgrounding. The SSH connection will die. The agent process will die or orphan. We must not let either of those things destroy the user's session.

**Approach:** ship a small companion daemon — `tether-bridge` — that the user installs on their dev box (Homebrew formula + a curl-pipe-bash for Linux). The daemon:

- Accepts an SSH-exec'd command like `tether-bridge attach <session-id>`.
- Owns the actual agent subprocess. When the app's SSH connection drops, the agent keeps running.
- Buffers all stdout from the agent (capped at e.g. 5MB per session) so a reconnecting client can replay missed output.
- Exposes session listing, creation, kill via the same protocol.

The app talks to `tether-bridge`, not to the agent directly. From the agent's perspective it's still stdio-attached to a normal client.

**On reconnect:** app SSH-execs `tether-bridge attach <id>`. Bridge replays the buffer (with a marker so the client can detect the replay boundary), then resumes live streaming.

The bridge is small (target <500 LoC Go) and is the most important infra piece in the product. Spec it carefully before writing the UI for it.

### 7.4 Backgrounding behavior

- On backgrounding: close SSH connection cleanly, persist session id and last-seen message offset to disk.
- On foregrounding: reopen SSH, reattach to session via bridge, request replay from last offset, merge into UI.
- Show a non-modal "reconnecting…" banner during the gap. Never lose a message because of a reconnect.

### 7.5 Security model

- All SSH credentials in Keychain with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`.
- No telemetry of message content, prompts, or file contents. Period. Crash reporting (Sentry) scrubs all strings from message events.
- No analytics SDK in beta. Anonymous install count via App Store Connect is sufficient.
- The app is a pipe. We are not in the path of the user's code or the model's output beyond rendering it.

## 8. UX requirements

- **First paint to usable chat:** < 2s on a warm connection.
- **Token latency:** first token from agent should render within 100ms of arrival on the wire. Measure end-to-end.
- **Tap targets:** all interactive elements ≥ 44pt.
- **Permission prompts:** must show command/path verbatim, never paraphrased. Syntax highlight. Default focus on Deny, not Allow.
- **Diffs:** unified format only (side-by-side does not work at phone width). Inline expand/collapse for context. Long-press a line to copy.
- **Hardware keyboard:** support iPad keyboard shortcuts (`⌘↩` to send, `⌘N` for new session, `Esc` to cancel agent). Beta-acceptable to leave unpolished on iPhone external keyboards.
- **Accessibility:** all interactive elements labeled for VoiceOver. Dynamic Type respected up to XL. No reliance on color alone for diff add/remove.

## 10. Success criteria for beta exit

- 20+ external TestFlight users have completed at least 5 sessions each.
- Crash-free session rate > 99%.
- Median time from app open (foreground) to receiving first new token in an existing session: < 3s.
- Zero reported incidents of credential leakage or message loss.
- Qualitative: at least 5 users say "I would pay for this" unprompted in feedback.

## 11. Open questions

These need resolution before or during the build, flagged for the engineer's attention:

1. **ACP version drift.** How do we want to gate features per-agent? Per-adapter capability flags vs. runtime probing. Lean adapter capability flags for beta.
2. **Bridge installation friction.** Homebrew is fine for Mac dev boxes. Linux servers need a one-liner. Windows is unsupported for beta — confirm.
3. **Push notifications for long-running agent tasks?** Probably post-beta but the bridge could enable it cleanly. Decide before bridge API freezes.
4. **Pricing model.** Not an engineering question, but it affects whether we need receipt validation in beta. Beta is free; defer.
5. **Tailscale-specific features.** Auto-discover machines on the tailnet? Nice to have, depends on whether their iOS SDK exposes what we need. Not blocking.

## 12. Risks

- **Apple review.** Low risk — SSH clients are an established category. Frame the app as "remote development client," lead with screenshots of diffs and code, not with "AI." Don't ship anything that downloads and executes code on-device.
- **ACP spec churn.** Medium risk. Mitigation: adapter layer, integration tests against pinned agent versions in CI.
- **iOS backgrounding fights us harder than expected.** Medium risk. The bridge design is the mitigation; if it fails we have no product. Validate end-to-end in Phase 3 before any further UI work.
- **Native SSH module reliability.** Medium risk. `NMSSH` is mature but not aggressively maintained. Budget time to fork or contribute fixes.
